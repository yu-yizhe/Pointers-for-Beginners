# 前言

> 《语言学纲要》给出了“语言”的官方定义：语言是人类社会约定俗成的，音义结合的全民性符号系统，是人类最重要的交际工具、思维工具。

> 无论是 C 语言，还是人们耳熟能详的 Python 语言，共属于一个概念叫“编程语言”。

> Claude Code、Codex、Cursor 风靡全球，真正实现了人们此前“一句话一个前端网页”的调侃。

早在很久以前，就有先知预测 AI 会重构人类社会。可“ AI 时代”真的到来了，却又是另一番不真实感。如果依旧觉得 AI 会和 2000 年的互联网一样是场泡沫，那请先以身入局，再下结论。

对绝大多数人而言，他们最关心的可能是 AI 是否会像互联网上不断翻炒的一样，会导致自己失业。到了今天这个时间点，我想很多人心中都有了答案。然而，出乎很多人意料的是，AI 最先替代的竟然会是视觉设计、程序员......但也是在意料之中的。

原因有以下三点：
> 01.工作相关数据开放度高
> 
> 02.工作结构化程度高
> 
> 03.结果容易量化

当然，并不是说 AI 会替代一个岗位中的所有人，至少目前在程序员这个行业当中，以淘汰低端程序员为主。但 AI 在 coding 这一块的发展速度以及潜力是有目共睹的，这也是为什么当 “vibe-coding” 一词被 Andrej Karpathy 提出后，越来越多编程小白也可以搓出自己的产品。

> Andrej Karpathy（2025-02-02 X 发文）：
> 
> _“There is a new kind of coding I call ‘vibe coding’, where you fully give in to the vibes, embrace exponentials, and forget that the code even exists.”_
> 
> “如今出现了一种全新的编程模式，我将其称作氛围感编程。全然沉浸创作状态，顺应指数级迭代逻辑，不必纠结代码本身。”

可随着越来越多人在互联网上鼓吹，“程序员的好日子到头了...今天我仿着 Apple.com vibe-coding 了一个网站...以后谁还学编程啊...”。我下意识感觉有点不对劲。真的不需要学编程了吗？思来想去之后，我给出一个我的答案：
> 不仅要学，比以前学的还要更深入

人们普遍的共识是：一个事物越稀缺，那么多数情况下就越有价值。

编程能力是互联网时代最重要的能力，甚至不需要加“之一”，难道其价值真的就一瞬间跌入谷底了？仔细思考就会发现 99% 的情况之下，随波逐流一定是错的（尽管多数人不愿承认）。举个简单的例子就是，一个散户，一边和多数散户一样去追涨杀跌，一边又幻想自己赚得盆满钵满，那最后能输得不只剩条裤衩子就不错了，够贴切了吧。

所以，如果人人都能够靠一句话就做出一个脚本，打造一个网页，甚至多说几嘴就搓出一个APP，那又有什么稀缺的呢？你可能在互联网上看到说：未来缺的不是解决问题的人，而是提出问题的人。但我想，一个想法，在如此透明的互联网上，你那怕憋着，憋坏了，说不定明天就有人 vibe-coding 出来并顺手开源了。这个世界并不缺那些天马行空的想法，毕竟不论你我，闲下来的时候也没少做白日梦吧。

再者，提到程序员，往往给人一种很有智慧感觉，这倒不全是刻板印象，因为掌握编程语言和没掌握编程语言确实是两个不同的世界。尴尬的是，这种微妙的区别只可意会，不可言传。而意会的唯一前提便是已经掌握了至少一门编程语言。因此，如果你目前确实是一名编程语言小白，我在这里也只能粗浅地点到为止，你想啊，可以多出一种看待问题的视角，何况还是个极具生产力的视角。（如果这还是没有点燃你心中那团火，不足以让你克服心中的恐惧，是的，就是恐惧，请勇敢承认，这很重要。推荐你可以从一门简单且非常实用的 Markdown 标记语言开始）。程序员的智力水平怎么说也不会太低，既然我们都能用明白，那程序员耍得不更溜？何况互联网行业对生产力工具的推行往往具有领先性，可能我们使用的都是他们玩剩下的。
> 人最可怕之处就是有傲慢之心，更可怕的是还不自知。
> 
> 请永远保持一颗学习之心，AI时代，怎么学都不够，难道不是吗？

既如此，哪怕你今天用 Cursor 写了个自动化脚本，明天又和 Claude Code 开发了一个微信小程序，其实也没什么好过分吹嘘的，这都应该感谢时代，庆幸自己身处这样一个大好的时代，而非成天抱怨环境有多差。因为，以前一个纯代码小白想要搭建一个自动化工作流，或许得先学习半年到一年 Python语言 ，然后在这个过程中脑海中的想法、完美的执行方案早就烟消云散了。而现在仅需要一两天的学习投入，一个此前仅仅是想想，但绝无可能实现的事情就那么被实现了。

有人可能要说了，我确实知道没有什么好过分吹嘘的，但还是想要发出来，刷波存在感。相信我，在这个时代，这不仅没有任何问题，反倒是明智的选择。当然，这份优越感实际上是建立在信息差的基础之上的。即你比别人先去了解了这个 AI工具，并使用了。然而随着时间的推移，大家最后都会用上 AI，这是必然的趋势。到那个时候你的优越感也便不复存在了。

那怎么办？可不就是很多“键盘侠”因为自己不敢学而劝大家都别学的编程吗？所以利用好 AI，以及这段时间差，深度掌握几门计算机语言（只要掌握了一门，后面学起来就是一通百通），把壁垒建立起来，赚到时间，然后接着学接着赚。

需要事先说明的是，本书并不会像传统 C 语言教科书一样去讲解相关知识。作为一本“小白书”，在相关知识点的语言表述上会遵循“能用一个字，绝不多用一个字”的原则，更追求表述的效率。

原因有以下三点：

01. C 语言自 1972 年由美国计算机科学家 Dennis Ritchie 发明后，至今已有 54 年的历史了。其相关的教材数不胜数，且早已更新迭代了不知到多少回了，无论是内容的完整度还是表述的精准性几乎都无可挑剔，因此如果你追求绝对官方的表述可以参照“微软官方 C 语言参考指南”（https://learn.microsoft.com/zh-cn/cpp/c-language）或是界内公认的 C 语言圣经《The C Programming Language》。

02.参考官方文档的确是非常不错的深刻理解 C 语言的一种方式，原因是当你看懂的时候，你就距离深度掌握不远了。挺尴尬的，对吗？这倒不是说官方教材写的不好，对于有一定基础且想要继续深造的人来说，肯定是再好不过的学习资料，但对于 C 语言小白来说并算不友好。有几个小白能够硬着头皮把教材啃完的？真的一个字一个字啃下来了，也有多半是一知半解，又有多少能够直面没搞懂的知识点，再反复阅读几遍的呢？答案是几乎没有！当然，在互联网还没有这么发达的时候，能够做到上述这些的人可能还会多一点。但现在，99% 的人可以说压根不具备长文本阅读能力，更别提是去理解了。多半是看了5分钟，就等不及抓起手机刷上一会，回过神来，一看时间，竟已经过了2个小时了。

03.事实上，人的大脑相比于文字，更喜欢可视化的视频形式。

> 美国教育心理学家 Richard E. Mayer 在《多媒体学习》中提到：
>
> “People learn better from words and pictures than from words alone.”
> “The human information processing system includes dual channels for visual/pictorial and auditory/verbal processing.”
>
> “人们从文字 + 图片中学得更好，远优于纯文字。”
> “人类信息加工系统包含双通道：视觉 / 图像通道与听觉 / 语言通道。”

也就是说，视频的“画面 + 声音”形式天然匹配大脑双通道，认知负荷最低、学习效率最高。

请试着回想一下自己看视频的时候，是不是几乎很难进行有效思考并提出疑问。

原因是，为了保证视频的留存率，一条视频的创作者对视频节奏的把控是相当厉害的，常见的套路就是业内熟知的开头一定要有3s黄金钩子，否则这个视频的留存率就会很低，因为统计表明。一个好的视频创作者，事实上完全可以做到让你完全跟着这个作者的思路和节奏，而不会有任何多余的你自己的念头。再加上你内心害怕去动手自己写代码，总想着看视频里讲解的老师的演示，就觉得自己看一遍和真正敲一遍是一样的。结果就是一个视频接着一个视频看，但又不肯动手跟着敲几行代码，于是看了后面忘了前面，等到要用了的时候大概率还是啥都不会，于是又啃哧啃吃跑回去看视频教程。接着又重复上述循环。当然不排除如此循环往复之后确实也可以掌握的还过得去，但一方面非常吃时间，很多人半途而废的原因就是因为自以为投入时间太多了，却迟迟没有看到正向反馈，于是就不敢学下去了。另一方面，不论是c语言还是其他任何编程语言，说到底就一个词“熟能生巧”。你写过100w行代码的就是比你只写了10w行代码的人更牛逼。因此，很抱歉的告诉你，学习编程语言想要速成是不可能的，编程语言能力的精进是无止境的，只有越写越熟练，越熟练越会写。

事实上在学习时，必要的文字阅读是不可或缺的。虽然阅读文字没有观看视频来的直接，但却可以激发你的思考，让你有提出问题的空间，而这是学习的关键，因为你真的在思考。这也是为什么，Claude Code 会选择以终端代码行的形式呈现，因为如此是效率最高的操作执行方式，尽管没有网页版看起来直观。

同样需要注意的就是：
> 理解 ≠ 掌握
> 理解 + 刻意练习 = 掌握

“刻意练习”一词源于 Anders Ericsson 和 Robert Pool 共创的《Peak: Secrets from the New Science of Expertise》一书。作者通过以下四句话精准解释了如何进行刻意练习。

> “Deliberate practice involves well-defined, specific goals, often aimed at some aspect of the overall performance; it is not aimed at some vague general improvement.”
> “目标必须定义明确、可执行、有时间限制，能直接指导练习内容。“
> 
> “Deliberate practice is intentional, meaning that it requires full attention and conscious action on the part of the practitioner.”
> “刻意练习需要全身心投入，注意力必须完全集中在任务上，不能分心、不能机械应付。”
>
> “Deliberate practice involves feedback and efforts to adjust in response to that feedback.”
> ”必须持续获得关于表现的反馈，清楚知道哪里做得好、哪里不足，以及如何改进。“
>
> “Deliberate practice takes place outside one’s comfort zone and requires near-maximal effort, which is generally not enjoyable.”
> “刻意练习发生在舒适区之外，要求不断尝试略高于当前能力的任务，长期停留在舒适区永远无法进步。”

即：刻意练习 = 特定目标 + 专注 + 反馈 + 走出舒适圈

几乎从小，我们就知道了“3W 1H”学习法，后来在英语的学习过程中也被不断提及。然而，越是简单的东西，往往人们越不愿意重视，应用起来却反而更难，以至于很多人都会致力于思考能否抽离出来一个学习新事物的万能模板。于是在走路了许多弯路之后，往往都会怅然一笑，留下一句“要是当时踏实一点，早点用‘3W 1H’学习法该有多好”。

所谓“3W 1H”学习法就是：
> What
> Why
> When
> How

因此，本书也将采用“3W 1H”学习法来向读者介绍 C 语言当中的指针部分。

可能会有读者疑问，为什么只写指针部分呢？

依旧是三点原因：

01.指针是 C 语言最重要的部分，掌握指针就约等于掌握了 C 语言。

02.其余部分难度较低，且是学习指针的基础，读者如果已经决定要学 C 语言了，自然不会有问题。大部分人都是因为无法理解运用指针，而半途而废的。

03.本书并不是 C 语言的教科书，只有教科书才会追求内容的绝对完整。本书的核心目标是帮助c指针初学者快速把握指针的逻辑链条，以及使用方法，以便可以迅速去实战，因为理论看再多也只是理论，学语言更重要的是使用，应该吧时间花在刀刃真正起决定性作用的地方。
