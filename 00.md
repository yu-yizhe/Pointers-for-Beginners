# 前言

> 《语言学纲要》给出了“语言”的官方定义：语言是人类社会约定俗成的，音义结合的全民性符号系统，是人类最重要的交际工具、思维工具。

> 无论是 C 语言，还是人们耳熟能详的 Python 语言，共属于一个概念叫“编程语言”。

> Claude Code、Codex、Cursor 风靡全球，真正实现了人们此前“一句话一个前端网页”的调侃。

早在很久以前，就有先知预测 AI 会重构人类社会。可“AI 时代”真的到来了，却又是另一番不真实感。如果依旧觉得 AI 会和 2000 年的互联网一样是场泡沫，那请先以身入局，再下结论。

对绝大多数人而言，他们最关心的可能是 AI 是否真的会像互联网上不断翻炒的一样，导致自己失业。到了今天这个时间点，我想很多人心中都有了答案。然而，出乎很多人意料的是，AI 最先替代的竟然会是视觉设计、程序员......但也是在意料之中的。

原因有以下三点：
> * 工作相关数据足够开放
>
> * 工作流程高度结构化
>
> * 工作结果容易被验证

当然，并不是说 AI 会替代一个岗位中的所有人，至少目前在程序员这个行业当中，以淘汰低端程序员为主。但 AI 在 coding 这一块的发展速度以及潜力是有目共睹的，这也是为什么当 “vibe coding” 一词被 Andrej Karpathy 提出后，越来越多编程小白也可以搓出自己的产品。

> Andrej Karpathy（2025-02-02 X 发文）：
> 
> _“There is a new kind of coding I call ‘vibe coding’, where you fully give in to the vibes, embrace exponentials, and forget that the code even exists.”_
> 
> “如今出现了一种全新的编程模式，我将其称作氛围感编程。全然沉浸创作状态，顺应指数级迭代逻辑，不必纠结代码本身。”

可随着越来越多人在互联网上鼓吹，“程序员的好日子到头了...今天我仿着 [Apple.com](https://www.apple.com) vibe coding 了一个网站...以后谁还学编程啊...”。我下意识感觉有点不对劲。真的不需要学编程了吗？思来想去之后，我给出一个我的答案：
> 不仅要学，比以前学的还要更深入

人们普遍的共识是：一个事物越稀缺，那么多数情况下就越有价值。

编程能力是互联网时代最重要的能力，甚至不需要加“之一”，难道其价值真的就一瞬间跌入谷底了？仔细思考就会发现 99% 的情况之下，随波逐流一定是错的（尽管多数人不愿承认）。举个简单的例子就是，一个散户，一边和多数散户一样去追涨杀跌，一边又幻想自己赚得盆满钵满，那最后能输得不只剩条裤衩子就不错了，够贴切了吧。

所以，如果人人都能够靠一句话就做出一个脚本，打造一个网页，甚至多说几嘴就搓出一个 APP，那又有什么稀缺的呢？你可能在互联网上看到说：未来缺的不是解决问题的人，而是提出问题的人。但我想，一个想法，在如此透明的互联网上，你那怕憋着，憋坏了，说不定明天就有人 vibe coding 出来并顺手开源了。这个世界并不缺那些天马行空的想法，毕竟不论你我，闲下来的时候也没少做白日梦吧。

再者，提到程序员，往往给人一种很有智慧感觉，这倒不全是刻板印象，因为掌握编程语言和没掌握编程语言的确是两个不同的世界。尴尬的是，这种微妙的区别只可意会，不可言传。而意会的唯一前提便是已经掌握了至少一门编程语言。因此，如果你目前确实是一名编程语言小白，我在这里也只能粗浅地点到为止，你想啊，可以多出一种看待问题的视角，何况还是个极具生产力的视角。（如果这还是没有点燃你心中那团火，不足以让你克服心中的恐惧，是的，就是恐惧，请勇敢承认，这很重要。推荐你可以从一门简单且非常实用的 Markdown 标记语言开始）。程序员的智力水平怎么说也不会太低，既然我们都能用明白，那程序员大概率耍得也不赖。何况互联网行业对生产力工具的推行可谓相当激进，可能我们使用的都是他们玩剩下的。
> 人最可怕之处就是有傲慢之心，更可怕的是还不自知。
> 
> 请永远保持一颗学习之心，AI时代，怎么学都不够，难道不是吗？

既如此，哪怕你今天用 Cursor 写了个自动化脚本，明天又和 Claude Code 开发了一个微信小程序，其实也没什么好过分吹嘘的，这都应该感谢时代，庆幸自己身处这样一个大好的时代，而非成天抱怨环境有多差。因为，以前一个纯代码小白想要搭建一个自动化工作流，或许得先学习半年到一年 Python 语言 ，然后在这个过程中，脑海里的想法、完美的执行方案早就烟消云散了。而现在仅需要一两天的学习投入，一个此前仅仅是想想，但绝无可能实现的事情就那么被实现了。

有人可能要说了，我确实知道没有什么好过分吹嘘的，但还是想要发出来，刷波存在感。相信我，在这个时代，这不仅没有任何问题，反倒是明智的选择。当然，这份优越感实际上是建立在信息差的基础之上的。即你比别人先去了解了这个 AI工具，并使用了。然而随着时间的推移，大家最后都会用上 AI，这是必然的趋势。到那个时候你的优越感也便不复存在了。

那怎么办？可不就是很多“胆小鬼”因为自己不敢学而劝大家都别学的编程吗？所以利用好 AI，以及这段时间差，深度掌握几门计算机语言（只要掌握了一门，后面学起来就是一通百通），把壁垒建立起来，赚到时间，然后接着学接着赚。

参考官方文档的确是非常不错的深刻理解 C 语言的一种方式，原因是当你看懂的时候，你就距离深度掌握不远了。挺尴尬的，对吗？这倒不是说官方教材写的不好，对于有一定基础且想要继续深造的人来说，肯定是再好不过的学习资料，但对于 C 语言小白来说并算不友好。有几个小白能够硬着头皮把教材啃完的？真的一个字一个字啃下来了，脑子里依旧是一团浆糊，又有多少能够直面没搞懂的知识点，再反复阅读几遍的呢？答案是几乎没有！当然，在互联网还没有这么发达的时候，能够做到上述这些的人可能还会多一点。但现在，99% 的人可以说压根不具备长文本阅读能力，更别提是去理解了。多半是看了5分钟，就等不及抓起手机刷上一会，回过神来，一看时间，竟已经过了2个小时了。

事实上，人的大脑相比于文字，更喜欢可视化的视频形式。

> 美国教育心理学家 Richard E. Mayer 在《多媒体学习》中提到：
>
> “People learn better from words and pictures than from words alone.”
> 
> “The human information processing system includes dual channels for visual/pictorial and auditory/verbal processing.”
>
> “人们从文字 + 图片中学得更好，远优于纯文字。”
> 
> “人类信息加工系统包含双通道：视觉 / 图像通道与听觉 / 语言通道。”

也就是说，视频的“画面 + 声音”形式天然匹配大脑双通道，认知负荷最低、学习效率最高。

请试着回想一下自己看视频的时候，是不是几乎很难进行有效思考并提出疑问。

原因是，为了保证视频的留存率，一条视频的创作者对视频节奏的把控是相当厉害的，常见的套路就是业内熟知的开头一定要有 3s 黄金钩子，否则这个视频的留存率就会很低，因为统计表明。一个好的视频创作者，事实上完全可以做到让你完全跟着这个作者的思路和节奏，而不会有任何多余的你自己的念头。再加上你内心害怕去动手自己写代码，总想着看视频里讲解的老师的演示，就觉得自己看一遍和真正敲一遍是一样的。结果就是一个视频接着一个视频看，但又不肯动手跟着敲几行代码，于是看了后面忘了前面，等到要用了的时候大概率还是啥都不会，于是又啃哧啃吃跑回去看视频教程。接着又重复上述循环。当然不排除如此循环往复之后确实也可以掌握的还过得去，但一方面非常吃时间，很多人半途而废的原因就是因为自以为投入时间太多了，却迟迟没有看到正向反馈，于是就不敢学下去了。另一方面，不论是c语言还是其他任何编程语言，说到底就一个词“熟能生巧”。你写过 100 万行代码的就是比你只写了 10 万行代码的人更牛逼。因此，很抱歉的告诉你，学习编程语言想要速成是不可能的，编程语言能力的精进是无止境的，只有越写越熟练，越熟练越会写。

事实上在学习时，必要的文字阅读是不可或缺的。虽然阅读文字没有观看视频来的直接，但却可以激发你的思考，让你有提出问题的空间，而这是学习的关键，因为你真的在思考。这也是为什么，Claude Code 会选择以终端代码行的形式呈现，因为如此是效率最高的操作执行方式，尽管没有网页版看起来直观。

同样需要注意的就是：
> 理解 ≠ 掌握
> 理解 + 刻意练习 = 掌握

“刻意练习”一词源于 Anders Ericsson 和 Robert Pool 共创的《Peak: Secrets from the New Science of Expertise》一书。作者通过以下四句话精准解释了如何进行刻意练习。

> “Deliberate practice involves well-defined, specific goals, often aimed at some aspect of the overall performance; it is not aimed at some vague general improvement.”
> “目标必须定义明确、可执行、有时间限制，能直接指导练习内容。“
> 
> “Deliberate practice is intentional, meaning that it requires full attention and conscious action on the part of the practitioner.”
> “刻意练习需要全身心投入，注意力必须完全集中在任务上，不能分心、不能机械应付。”
>
> “Deliberate practice involves feedback and efforts to adjust in response to that feedback.”
> ”必须持续获得关于表现的反馈，清楚知道哪里做得好、哪里不足，以及如何改进。“
>
> “Deliberate practice takes place outside one’s comfort zone and requires near-maximal effort, which is generally not enjoyable.”
> “刻意练习发生在舒适区之外，要求不断尝试略高于当前能力的任务，长期停留在舒适区永远无法进步。”

即：刻意练习 = 特定目标 + 专注 + 反馈 + 走出舒适圈

几乎从小，我们就知道了“3W 1H”学习法，后来在英语的学习过程中也被不断提及。然而，越是简单的东西，往往人们越不愿意重视，应用起来却反而更难，以至于很多人都会致力于思考能否抽离出来一个学习新事物的万能模板。于是在走路了许多弯路之后，往往都会怅然一笑，留下一句“要是当时踏实一点，早点用‘3W 1H’学习法该有多好”。

所谓“3W 1H”学习法就是：
> What
> 
> Why
>
> When
>
> How

因此，本书也将采用“3W 1H”学习法来向读者介绍 C 语言当中的指针部分。

其余部分难度较低，且是学习指针的基础，读者如果已经决定要学 C 语言了，自然不会有问题。大部分人都是因为无法理解运用指针，而半途而废的。

本书并不是 C 语言的教科书，只有教科书才会追求内容的绝对完整。本书的核心目标是帮助c指针初学者快速把握指针的逻辑链条，以及使用方法，以便可以迅速去实战，因为理论看再多也只是理论，学语言更重要的是使用，应该吧时间花在刀刃真正起决定性作用的地方。

也正因如此，视频天然比文字更容易吸引人。但问题在于：视频提升了信息的吸收效率，却未必提升了思考效率。

请试着回想一下，当你在刷视频教程时，是不是很少真正停下来思考，更少主动提出问题。

原因很简单。为了提高视频的留存率，一个成熟的视频创作者，往往会极其精准地控制你的注意力节奏。比如业内熟知的“3 秒黄金钩子”：如果开头几秒无法抓住用户，视频的数据就会迅速下滑。因此，大量视频内容的本质，其实是在不断推动你继续看下去。好的视频创作者，甚至可以让你完全跟着他的节奏走，而不给你留下太多独立思考的空间。于是很多人会进入一种典型循环：看视频 -> 觉得自己懂了 -> 不愿动手 -> 很快忘记 -> 再回去看视频。尤其是学习编程时，这种现象尤为常见，总觉得看老师敲一遍，和自己写一遍差不多，再加上内心本就害怕，于是就更不愿动手写代码了。但
真正让人进步的，从来不是“看懂”，而是“写过”。编程语言本质上只有四个字：熟能生巧。写过 100 万行代码的人，就是会比只写过 10 万行代码的人更熟练。因此，很遗憾的是编程能力不存在真正意义上的“速成”，且这个能力的精进是无止境的，只能通过不断地刻意练习，一点点积累。

“刻意练习”一词源于 Anders Ericsson 与 Robert Pool 合著的《Peak: Secrets from the New Science of Expertise》一书。作者用四句话精准概括了“刻意练习”的核心：

> “Deliberate practice involves well-defined, specific goals, often aimed at some aspect of the overall performance; it is not aimed at some vague general improvement.”
> “目标必须定义明确、可执行，并能够直接指导练习内容，而不是停留在模糊的‘我想变强’。”
> 
> “Deliberate practice is intentional, meaning that it requires full attention and conscious action on the part of the practitioner.”
> “刻意练习需要全身心投入，注意力必须高度集中，而不是机械重复。”
>
> “Deliberate practice involves feedback and efforts to adjust in response to that feedback.”
> “必须持续获得反馈，知道哪里做得对、哪里不足，并不断调整。”
>
> “Deliberate practice takes place outside one’s comfort zone and requires near-maximal effort, which is generally not enjoyable.”
> “刻意练习发生在舒适区之外，它要求你长期挑战略高于当前能力的任务。”

即：刻意练习 = 明确目标 + 高度专注 + 持续反馈 + 走出舒适圈

这也是为什么很多人看了大量视频，却依旧学不会编程。因为视频，本质上是一种更符合人性的学习方式。它更轻松、更直观、更容易获得即时反馈。你几乎不需要停下来思考，只需要顺着视频节奏不断往下看即可。可问题在于，舒服，并不等于有效。真正的学习，往往发生在“不舒服”的地方。阅读文字，相比看视频，更容易让人产生理解阻力，这恰恰会迫使你离开这种舒适区。也正因如此，文字会迫使你停下来，并开始主动思考：为什么代码这样写？为什么这里这样设计？有没有别的实现方式？...提出问题，本身就是学习真正开始的标志。Anthropic 的 Claude Code 选择以终端代码行的形式呈现，也是因为对于真正高频、高强度的操作而言，简洁、可控、可思考，往往比直观更重要。

需要事先说明的是，本书并不会像传统 C 语言教科书一样去讲解相关知识。作为一本“小白书”，在知识点的表述上会遵循“能用一个字，绝不多用一个字”的原则，更追求表述的效率。






可能会有读者疑问，为什么只写指针部分呢？

指针是 C 语言最核心、也是最难理解的部分。某种程度上，掌握了指针，也就等于真正掌握了 C 语言。至于其余部分，虽然同样重要，但整体难度并不高，并且本身就是学习指针的基础。既然读者已经决定学习 C 语言，那么这些内容大概率都不会是问题。真正让大量初学者半途而废的是无法理解与运用指针。因此，本书并不会试图覆盖 C 语言中的所有内容，只有教科书，才会追求知识体系的系统完整性。

因为理论看得再多，也终究只是理论。学习编程语言，更重要的始终是“使用”。所以相比不断堆积知识点，本书更希望：帮助读者快速掌握关于指针的逻辑链条和实际运用方式，以便尽快进入实战，把时间真正花在决定性的问题上。

同时也需要提前说明，本书并不会采用传统 C 语言教材的讲述方式。作为一本面向初学者的“小白书”，本书在知识表达上会尽可能遵循一个原则：能一句话讲明白，就绝不讲两句。因为相比“讲得全面”，本书更在意的是：讲得足够清楚、足够高效。
